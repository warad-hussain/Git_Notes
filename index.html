Git & GitHub

What is Git & GitHub?

let's use an analogy to understand it perfectly:

Imagine you're working on a big puzzle, and you want a way to save your progress and also share it with friends.

Git is like a magic photo camera. Every time you add some pieces to the puzzle and feel good about it, you take a picture with this camera. If you ever mess up, you can always go back to any of your previous photos and start from there. This camera helps you keep track of all the changes you made to your puzzle.

GitHub is like a public gallery where you can hang your puzzle photos for others to see. Friends can come, check out your puzzle, and even help by adding pieces. They can also show you their puzzles and get your help. It's a place where everyone shares and collaborates on their projects.

So, Git helps you manage and track your project changes, while GitHub is a platform where these projects are shared and collaborated on with others.

GitHub Common Commands

The selected text lists common commands for using Git, a version control system for software development. The commands are organized by category, including setup, making changes, branching, updating and undoing, and miscellaneous. The commands listed include initializing a new Git repository, staging and committing changes, creating and switching branches, fetching and merging changes from a remote repository, and displaying the commit history and changes made to files.

Setup

  • git init: Initializes a new Git repository in the current directory.
  • git clone [repository]: Creates a local copy of an existing repository.
  • git config --global user.name "[name]": Sets the name associated with your Git commits.
  • git config --global user.email "[email address]": Sets the email address associated with your Git commits.
  • git remote add origin [repository URL]: Links the local repository to the remote repository.

Making Changes

  • git status: Displays the status of the Git repository.
  • git add [file]: Stages a file to be committed.
  • git add .: Stages all changes in the working directory to be committed.
  • git commit -m "[commit message]": Commits changes to the repository with a message.
  • git push: Uploads local changes to the remote repository.

Branching

  • git branch: Lists all branches in the repository.
  • git branch [branch name]: Creates a new branch.
  • git checkout [branch name]: Switches to a different branch.
  • git merge [branch name]: Merges a branch into the current branch.
  • git branch -d [branch name] Delete a branch
  • git push origin --delete [branch name] Delete a remote branch

Updating and Undoing

  • git pull: Fetches and merges changes from the remote repository.
  • git reset: Unstages changes in the staging area.
  • git revert [commit hash]: Undoes a commit.
  • git reset --hard [commit hash]: Resets to a specific commit.

Miscellaneous

  • git log: Displays the commit history of the repository.
  • git log --summaryView changes (detailed)
  • git log --onelineView changes (briefly)
  • git diff [file]: Displays changes made to a file.
  • git diff [source branch] [target branch]Preview changes before merging
  • git remote -v: Displays the remote repositories connected to the local repository.
  • git rm -r [file-name.txt] Remove a file (or folder)

Important Concepts

Understand Branches :

Git branches are an important part of version control in software development. A branch is like a separate line of development within a project. It allows multiple developers to work on different parts of the codebase without interfering with each other.

Imagine a tree trunk with branches coming off of it. Each branch represents a different version of the codebase. Developers can work on their own branch and make changes without affecting the master branch. When they are ready, they can merge their changes back into the master branch.

Here is an example diagram of Git branches:

          A---B---C  (master)
               \\\\
                D---E  (feature)

In this diagram, the master branch contains three commits (A, B, and C), while the feature branch contains two commits (D and E). The commit marked as C is the "head" of the master branch, while the commit marked as E is the "head" of the feature branch.

To switch between branches, you can use the git checkout command followed by the name of the branch you want to switch to. For example, to switch to the feature branch, you would use the command git checkout feature.

Push and Pull Request :

Push

When you have made changes to your local repository and want to upload those changes to the remote repository, you need to push them. Here is the process for pushing changes:

  1. Make changes to your local repository using Git commands (such as git add and git commit).
  1. Use the git push command to upload your changes to the remote repository.
  1. If you are pushing changes to a branch that does not exist on the remote repository, use the git push --set-upstream origin [branch name] command to create the branch on the remote repository.

Pull Request

A pull request is a way to propose changes to a repository hosted on GitHub. Here is the process for creating a pull request:

  1. Fork the repository you want to make changes to.
  1. Clone the forked repository to your local machine using the git clone [repository URL] command.
  1. Create a new branch for your changes using the git branch [branch name] command and switch to that branch using the git checkout [branch name] command.
  1. Make changes to the files in your local repository using Git commands (such as git add and git commit).
  1. Push your changes to the remote repository using the git push command.
  1. Go to the repository on GitHub and click the "New pull request" button.
  1. Select the branch you made your changes on as the "compare" branch and the branch you want to merge your changes into as the "base" branch.
  1. Add a title and description for your pull request and click the "Create pull request" button.

After you create a pull request, other contributors to the repository can review your changes and leave comments. Once the changes are approved by Owner, they can be merged into the base branch.

Conflicts :

In Git, a conflict occurs when two or more people are editing the same file at the same time, and their changes conflict with each other. When two people try to merge their changes together, Git may not be able to do so automatically because it doesn't know which version to use.

To resolve a conflict, the conflicting parties need to manually edit the file to remove the conflicting changes. Once the file has been edited, it needs to be committed to the repository to resolve the conflict.

Conflicts can be avoided by communicating with other contributors and coordinating changes. It's also important to regularly pull changes from the remote repository to ensure that you are working with the latest version of the codebase.

Note: To avoid conflicts always pull that branch in which you are working to work in up to date state.

GitHub Issues :

GitHub Issues is like a to-do list for software projects on GitHub. People can report problems, suggest new features, or discuss changes they'd like to see. It helps developers keep track of what needs to be done, what's in progress, and what's already finished.
